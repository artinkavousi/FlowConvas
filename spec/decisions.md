# Decisions — ARTINOS

> ADR-lite log. Append only; never rewrite history. Each: context → decision → consequences.
> Format: ADR-N (date) · status.

---

## ADR-1 (2026-06-22) · accepted — ARTINOS v1 is single-user, local-first

**Context.** ARTINOS-PRD §21 left open whether ARTINOS is single-user/local-first or multi-user/collaborative. This shapes the entire product (auth, backend, sync vs none).
**Decision.** v1 is single-user, local-first: runs locally, one creator, no accounts/auth/cloud sync. Multi-user is explicitly deferred.
**Consequences.** No backend or auth work in v1 (PRD NG-2). Matches the "Studio first, packages later" principle. Multi-user becomes a future, additive concern, not a v1 constraint.

## ADR-2 (2026-06-22) · accepted — File-based registry, no database

**Context.** ARTINOS-PRD §21 open question on the persistence layer (files, DB, or both).
**Decision.** The registry is local file-based metadata: one co-located entry per module, discovered at runtime; files are the source of truth. No DB or server in v1.
**Consequences.** Git-friendly, agent-readable, zero infra. Fast cross-module search at scale (thousands of modules) is not optimized in v1; a local index/SQLite is a Post-MVP option if needed. Drives ADR-5 (discovery) and ADR-8 (validation).

## ADR-3 (2026-06-22) · accepted — "Agent-operable" v1 = structured metadata + documented workflow

**Context.** ARTINOS-PRD §21 / §10 leave open how deep agent operability goes in v1 (metadata only vs MCP/tool endpoints vs bespoke runtime).
**Decision.** v1 ships machine-readable registry entries + `agentNotes` + a discoverable index, and the converter is a documented agent workflow/skill — not a custom runtime or sandbox. MCP/tool endpoints (PRD FR-21) are Post-MVP.
**Consequences.** No sandbox/runtime build in v1 (PRD NG-5). The agent operates the studio by reading entries and following `spec/converter-workflow.md` + `STUDIO/AGENTS.md`. Programmatic tool-driven operation is a later, additive layer.

## ADR-4 (2026-06-22) · accepted — Workspace wiring: npm workspaces + Vite source alias  (resolves PRD Q-3)

**Context.** The Studio in `STUDIO/` must consume PANELFLOW so changes flow without npm publishing (PRD FR-2). Options: `file:` dep on PANELFLOW's built `dist` (needs a watch build, loses HMR); path alias only (no dep hoisting); workspaces + alias to source.
**Decision.** Add a root `package.json` with npm workspaces `["PANELFLOW","STUDIO"]`, and in the Studio's Vite config alias `@artinos/panelflow` → `../PANELFLOW/src/export.ts` (and `@` → `../PANELFLOW/src`). The Studio consumes PANELFLOW **source**.
**Consequences.** True HMR with one dev command; PANELFLOW edits reflect instantly. PANELFLOW's runtime deps must hoist (workspaces handle this) and the Studio's Tailwind v4 must scan PANELFLOW source via `@source "../PANELFLOW/src"` (Risk R-1, verified in T-2). Documented fallback: consume `dist` + `build:lib --watch`.

## ADR-5 (2026-06-22) · accepted — PANELFLOW `ComponentSchema` is the canonical control schema  (resolves PRD Q-2)

**Context.** Showcases need parameter controls; PANELFLOW already has `ComponentSchema` + `registerComponent` + `AutoGeneratedPanel`. Should ARTINOS define its own schema or reuse PANELFLOW's?
**Decision.** Reuse PANELFLOW's `ComponentSchema` directly as the canonical control schema. The `ArtinosModule` registry entry embeds it as `schema`. Showcases call `registerComponent(module.schema)` and render `AutoGeneratedPanel`.
**Consequences.** No translation layer; showcase controls are driven by the already-built auto-panel. `schema.id` must equal `module.id` so bridge values key correctly (enforced by the validator, ADR-8). Couples ARTINOS modules to PANELFLOW's schema type — acceptable since PANELFLOW is the designated foundation.

## ADR-6 (2026-06-22) · accepted — Per-module entry + `import.meta.glob` discovery  (resolves PRD Q-1)

**Context.** How to store and load registry entries (one manifest vs per-module files; codegen vs runtime).
**Decision.** Each module is a folder `STUDIO/src/modules/<id>/` with `<id>.module.ts` default-exporting an `ArtinosModule`. `registry.ts` builds the index at runtime via `import.meta.glob('../modules/*/*.module.{ts,tsx}', { eager: true })`. No codegen step.
**Consequences.** Adding a module = adding a folder; the registry picks it up automatically. Bundles all modules eagerly (fine at MVP scale; lazy/`{ eager:false }` is a Post-MVP option for large libraries). Each entry must have a `default` export (validated, ADR-8).

## ADR-7 (2026-06-22) · accepted — Converter is a documented workflow + scaffold script  (resolves PRD Q-4)

**Context.** Where the converter lives and how it is invoked.
**Decision.** The converter is `spec/converter-workflow.md` (the agent-followed procedure mapping ARTINOS-PRD §10/§17/§18) plus `STUDIO/scripts/new-module.ts` (stamps the module-folder boilerplate). Conversion fidelity follows AGENTS.md §4 (port directly, preserve identity).
**Consequences.** No bespoke converter runtime in v1 (ADR-3). The agent does the conversion; the script removes boilerplate; the validator (ADR-8) gates the result. The external-conversion proof depends on a source being provided (ADR-9).

## ADR-8 (2026-06-22) · accepted — Validation rules & the library-sync gate  (resolves PRD Q-5)

**Context.** PRD metrics M-2/M-5 and FR-8/FR-20 need concrete definitions of "complete entry" and "passing showcase."
**Decision.** `STUDIO/scripts/check-registry.ts` is the gate. A **complete entry** = all required `ArtinosModule` fields present and non-empty, `schema.id === module.id`, `sourcePath` resolves on disk, every `schema.parameters` item has `key/label/type/default`, and no duplicate ids. A **passing showcase** = builds, the preview mounts with zero console errors, and controls drive the preview. The script prints a per-module table and exits non-zero on any failure.
**Consequences.** "Done" for any module = `check-registry` green + build/preview/console proof (folds into the DoD in `STUDIO/AGENTS.md`). Library drift is caught mechanically (FR-20).

## ADR-9 (2026-06-22) · accepted — Seed set, and the external-conversion proof is gated  (resolves PRD Q-6)

**Context.** PRD M-6 wants ≥12 modules over a quarter and FR-16 wants one real external conversion. The WebGPU Fluid Simulation source named in ARTINOS-PRD §17 (`WebGpu-Fluid-Simulation-master`) is **not present anywhere in the repo** (verified at planning).
**Decision.** The first three registered modules are PANELFLOW's `GooeySlider`, `BubbleRatingSlider`, `ElasticMenu` (guaranteed available, already premium) — these prove the registry/gallery/showcase loop. The external-conversion proof (T-14) is **gated**: the agent first locates the source and, if absent, reports BLOCKED per AGENTS.md §4 and asks the user to provide it or confirm an alternative available Three.js/R3F/ShaderToy source.
**Consequences.** The MVP loop is fully demonstrable without the external source. FR-16's external half may end the MVP in a documented BLOCKED state until the user supplies the source — acceptable per AGENTS.md.

## ADR-10 (2026-06-22) · accepted — No router; navigation via a tiny store

**Context.** Showcases are described as "routes" (PRD FR-10). Add `react-router-dom` or use internal state?
**Decision.** Use a minimal `useStudioStore` (`view: 'gallery' | 'showcase'`, `activeModuleId`); no router dependency in v1.
**Consequences.** Fewer deps, simpler MVP (local-first, single-user needs no shareable URLs). Shareable/deep-link URLs are a clean Post-MVP add (swap the store for a router without touching modules).

## ADR-12 (2026-06-22) · accepted — Studio must declare PANELFLOW's runtime deps + force a single React (execution finding)

**Context.** ADR-4 assumed npm workspaces would hoist PANELFLOW's deps for the Studio. In practice PANELFLOW kept its own populated `node_modules` + lockfile, so its deps did **not** hoist to root. Two concrete failures during T-3/T-7: (a) `tsc`/Vite couldn't resolve `zustand` from the Studio's own files; (b) consuming PANELFLOW **source** loaded React twice, breaking `useSyncExternalStore` ("getSnapshot should be cached" + black-screen render loop).
**Decision.** The Studio declares the runtime deps its own code imports (`zustand`, `lucide-react`, `clsx`) and forces a single React copy via explicit Vite aliases (`react`/`react-dom` → the one root install) + `dedupe` + `optimizeDeps.include`. Added `src/vite-env.d.ts` for `vite/client` types (`import.meta.glob`).
**Consequences.** `vite.config.ts` carries explicit React aliases beyond the plan's `dedupe`-only note. Future modules that import other PANELFLOW runtime deps directly must add them to the Studio's deps too. Fallback (consume PANELFLOW `dist`) remains documented in ADR-4.

## ADR-13 (2026-06-22) · accepted — Module previews select the raw bridge slice, default outside the selector (execution finding)

**Context.** A preview reading `useBridgeStore((s) => s.componentValues[id] || {})` returns a fresh `{}` each render before the bridge is initialized, tripping the same getSnapshot loop (PANELFLOW's own auto-panel avoids it only because its bridge is pre-initialized).
**Decision.** Previews select the raw (possibly `undefined`) slice — `useBridgeStore((s) => s.componentValues[id])` — and apply fallback defaults *outside* the selector. The T-10/T-11 seed modules mirror this fixed pattern.
**Consequences.** Previews render safely standalone (before any `registerComponent`), so the gallery→showcase path works even ahead of T-8's bridge initialization.

## ADR-11 (2026-06-22) · accepted — Showcase controls reuse PANELFLOW's `AutoGeneratedPanel`

**Context.** The showcase needs a controls UI. Build one or reuse PANELFLOW's?
**Decision.** Render PANELFLOW's exported `AutoGeneratedPanel` inline for the controls section; `registerComponent` also surfaces the panel in the dock rail / command palette.
**Consequences.** Zero new control UI in ARTINOS (reuse over rebuild). Control look/behavior is governed by PANELFLOW; any control improvements happen in PANELFLOW, not the Studio. **Superseded by ADR-14** — controls now live only in the dock, not inline in a page.

## ADR-14 (2026-06-22) · accepted — Viewport = live stage; controls live in the dock  (supersedes ADR-10/11)

**Context.** The Studio rendered a `Gallery`/`Showcase` *documentation page* inside the PANELFLOW viewport, duplicating controls inline while the editor dock showed an unrelated node graph. The component never truly "ran in the viewport," and PANELFLOW's panel system was bypassed.
**Decision.** The viewport renders ONLY the active module's live preview (fullscreen `StudioViewport` → `PreviewStage`). Selecting a module sets `studio-store.activeModuleId`; a `useActiveModule` hook (in `ArtinosStudio.tsx`) registers its schema, seeds bridge defaults, and opens the auto-generated control panel + an `Info` panel in the dock. The old `Gallery.tsx`/`Showcase.tsx` pages are deleted; their browse/info UI moves into `library.panel.tsx` and `module-info.panel.tsx` (PANELFLOW dock panels). Navigation stays router-free (ADR-10's store, reshaped to `activeModuleId`).
**Consequences.** Controls and the running component are no longer duplicated or disconnected. The Library/Info panels are ordinary PANELFLOW panels. `Gallery`/`Showcase` and the `view` state are gone.

## ADR-15 (2026-06-22) · accepted — Auto-generated control panels render with tweakpane_frost

**Context.** `generatePanelFromSchema` wired up the plain-HTML `AutoGeneratedPanel`; the tweakpane_frost panel (`FrostPanePanel`) existed and was exported but unused, despite tweakpane_frost being the chosen control language.
**Decision.** `generatePanelFromSchema` renders `FrostPanePanel`. `FrostPanePanel` is schema-driven: `ParameterDef.group` → Tweakpane folders, `schema.modifiers` → folders, edits write the shared `useBridgeStore` (keyed by `schema.id`), and external bridge changes (presets) refresh the pane in place. `AutoGeneratedPanel` stays exported as a fallback.
**Consequences.** All component control panels are tweakpane_frost and derived purely from schema + bridge — copy-paste portable (ADR-5). Tweakpane's bundled plugins emit a benign `onChange` deprecation warning.

## ADR-16 (2026-06-22) · accepted — The dock is a multi-panel host; the node graph is an optional panel

**Context.** The editor dock hardcoded the node-graph canvas as its centerpiece with a single slide-out side panel. The control-panel-first model needs the dock to host several control panels at once.
**Decision.** `panel-store` gains `openPanelIds` + `activePanelId` (+ `openPanel`/`closePanel`/`focusPanel`). `editor-dock` renders open panels as a **tab strip** (one active at a time), keeping all existing chrome (toolbar, scene controls, perf stats, window/dock modes, rail, resize). The graph becomes a regular panel (`graph.panel.tsx`, owning its own `ReactFlowProvider`) opened from the rail. Side-by-side tiling (dockview) is deferred.
**Consequences.** Many control panels coexist as tabs; the graph is opt-in, not the default. `IconRail` reads `activePanelId`/`openPanelIds`. The legacy `dockedPanelId` remains only so graph os-panel nodes keep working.

## ADR-17 (2026-06-22) · accepted — PANELFLOW is a real local package and controls target instances

**Context.** The Studio imported `@artinos/panelflow` through Vite source aliases, but did not declare the package dependency. PANELFLOW also shipped raw `src/lib/tweakpane_frost` internals in `npm pack --dry-run`, and the control bridge keyed values only by `schema.id`, which would make multiple copies of the same component overwrite each other.
**Decision.** `STUDIO` declares `@artinos/panelflow` as `file:../PANELFLOW` while retaining source aliases for dev HMR. PANELFLOW ships only built `dist` artifacts and exposes the Frost runtime through a built `./frost` subpath. The control engine now supports `registerComponentInstance` / `unregisterComponentInstance`; generated panels can bind either legacy schema buckets or concrete instance ids.
**Consequences.** ARTINOS consumes PANELFLOW like a node_modules package but can still edit it live during development. Future multi-component projects should register concrete instances, not only schemas. Library packing no longer leaks TSX/source internals; heavy Tweakpane/Three payloads are split into intentional lazy chunks.

## ADR-18 (2026-06-23) · accepted — Fluid module re-expanded to full source fidelity (supersedes the T-14 strip)

**Context.** Task T-14 ported the WebGPU fluid but **stripped** audio-reactivity, the emitter system, presets, and the performance scaler to keep the MVP module minimal. ARTINOS-PRD §17 names exactly those systems (emitter controls, audio-reactive options, performance settings) as the deliverable; FR-15 mandates preserving the source's behavior, audio, and interactions — so the stripped version under-delivered the flagship conversion.
**Decision.** Restore the fluid to full source fidelity: `sim/audio/*` (beat tracking, feature extraction, modulation matrix), `sim/emitters/*` (15+ emitter types + samplers), `sim/presets/*` (18 built-in presets), `sim/performance/QualityScaler.js`, and `sim/params.ts` (~111 controls). Fidelity over minimalism for the designated proof-of-loop module.
**Consequences.** Heavier bundle (mitigated by lazy chunks per ADR-17 — `WebGPUFluidPreview` and `three`/`tweakpane` split out). The fluid is now the reference for what a "full-fidelity conversion" means (FR-15/§17). check-registry still green; build still green.

## ADR-19 (2026-06-23) · accepted — Completion plan governs v0.3 → v1; adopts D-A…D-F

**Context.** The 2026-06-23 ground-truth audit (lint/build/check-registry/live-preview, not doc-trust) found the MVP loop working but the PRD vision ~40-45% met: library is 4 modules vs ≥12 target, no agent panel / graph-node exposure / MCP endpoints / public website, and the spec docs had drifted from the panel architecture.
**Decision.** Adopt `spec/plan-completion.md` as the governing plan for completion, with design decisions D-A (graph nodes via a generated `moduleToNode` adapter), D-B (agent panel reads the registry, no embedded LLM), D-C (MCP wraps existing scripts, files stay canonical), D-D (console panel is a thin mirror, not a framework), D-E (Website is a read-only static reader of the registry), D-F (library breadth grows by converter discipline, one category at a time). Tasks tracked in `spec/tasks-completion.md` (C-1…C-19).
**Consequences.** Post-MVP FRs (FR-9 graph nodes, FR-21 MCP, FR-22 website, FR-23 promotion) move into scope. No new persistence or bespoke runtime — every new agent surface wraps the existing file-based registry (preserves ADR-2/3).

## ADR-20 (2026-06-23) · accepted — Module→package promotion is gated, criteria-driven, and reversible (FR-23)

**Context.** PRD §7 says "Studio first, packages later" and §21 left open the concrete criteria that trigger promoting a module out of the Studio into its own package.
**Decision.** Adopt `spec/promotion-workflow.md`: a module promotes only when ALL of — ≥3 real-project reuse, ≥4-week stable API, framework-agnostic core (no PANELFLOW/bridge hard-dep), `check-registry` green, minimal owned deps, and a second-maintainer signal — hold. Promotion extracts the core to `packages/<id>/` (`@artinos/<id>`), keeps the Studio module as the showcase (re-pointed `sourcePath`/`dependencies`), adds the workspace, re-validates, and records an ADR. De-promotion (fold back to Studio source) is allowed to keep the bar honest.
**Consequences.** Premature packaging stays a non-goal (PRD §19); the registry/showcase/Agent/graph/MCP surfaces are unchanged by promotion since they read the same `ArtinosModule` entry. `packages/` is introduced only when the first module actually qualifies.
